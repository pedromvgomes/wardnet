# Wardnet 0.2.0

The first signed, publicly-verifiable release of Wardnet — a self-hosted network privacy gateway.

Wardnet sits alongside your existing home or small-office router and acts as the warden of every device's connection to the internet: network-wide ad and tracker blocking at the DNS level, per-device VPN routing through WireGuard tunnels, and full control from a local web dashboard. Think of it as a Pi-hole replacement with per-device VPN routing built in.

Devices that can't run VPN software themselves — smart TVs, consoles, IoT — get the same protection at the gateway.

This release establishes the foundation for everything that follows. It's daily-driven on a single Pi at home, but expect to read the source occasionally if you hit rough edges.

## What you can do with Wardnet today

### Network-wide ad and tracker blocking (Pi-hole replacement)

- Built-in DNS server with authoritative recursive resolution via [hickory-dns](https://github.com/hickory-dns/hickory-dns).
- Cron-scheduled blocklist refresh — StevenBlack, OISD, AdGuard lists, or bring your own. Bring your existing Pi-hole lists straight over.
- **Allowlist** and **custom filter rules** for surgical exceptions.
- Configurable upstream DNS servers with DoT / DoH / plain UDP support.
- In-memory cache with configurable TTL bounds and flush-on-demand.
- Applies to every device on the LAN, regardless of routing policy.

### Per-device VPN routing

- **WireGuard tunnel management** — add tunnels by importing a `.conf` file or by provisioning through a VPN provider integration. Interfaces come up on demand and tear down after an idle timeout.
- **NordVPN provider integration** — validate credentials, browse countries and servers, and set up a tunnel with one click. Supports dedicated-IP servers via direct hostname.
- **Per-device policies** — route each known device through a specific tunnel, directly to the internet, or through the network default. Policies are applied via `ip rule` + nftables with automatic reconciliation on startup.
- **Admin / self-service split** — admins set policies for any device; end-users can see and change their own policy through an unauthenticated self-service page (identified by source IP). Admin-locked devices are protected from user changes.
- **DNS leak prevention** — when a device is routed through a tunnel, port 53 traffic is DNAT'd to the tunnel's own DNS so queries stay inside the encrypted path.

### Built-in DHCP server

- Full lease lifecycle: `DISCOVER` / `REQUEST` / `RELEASE` with pool allocation, expiry, and audit log.
- Static MAC-to-IP reservations with hostname and description.
- Gateway IP auto-detected from the LAN interface; upstream DNS and pool range configurable from the web UI.
- Toggleable — off by default; leave your existing DHCP source alone until you're ready to hand over.

### Device discovery

- Background ARP scan + passive packet capture identifies devices on the LAN.
- MAC OUI lookup against the full IEEE MA-L database (~39k entries, embedded at build time) for vendor hints.
- Randomised-MAC detection flags locally-administered addresses (phones using MAC rotation).
- Hostname resolution via reverse-DNS and mDNS.
- Devices auto-track their last-seen IP and survive DHCP re-assignments.

### Web UI

- React 19 + TypeScript, served from the single daemon binary via `rust-embed`.
- **First-run setup wizard** creates the initial admin account.
- **Dashboard** with live system status, recent errors, and log viewer (WebSocket streaming).
- **Devices** — list, detail, edit name/type, set routing policy, lock/unlock admin control.
- **Tunnels** — create (manual or provider-provisioned), view status, delete.
- **DHCP** — pool config, leases, reservations, status.
- **DNS / Ad blocking** — resolver config, blocklists, allowlist, custom rules.
- **Settings** — version, uptime, and account management.
- Dark / light mode following the system preference.

### API and CLI

- **REST + WebSocket HTTP API** — every UI feature is driven through it. Session cookie or API-key auth for admin endpoints; a small set of self-service endpoints identify the caller by source IP.
- **`@wardnet/js` SDK** — zero-dependency TypeScript client that works in browsers and Node 18+. Mirrors the daemon API DTOs.
- **`wctl` CLI** — scaffolded with subcommands for status, devices, and tunnels. Full implementations land in a future release.

### Observability

- **OpenTelemetry** — OTLP export of traces, structured logs, and metrics (opt-in). Hierarchical tracing spans rooted at `wardnetd{version=...}` so every log line carries the daemon version and a correlation ID.
- **Pyroscope continuous profiling** — opt-in CPU profiling for performance work.
- **Structured JSON logs** to disk with rotation, plus an in-memory ring buffer of recent errors surfaced in the UI.

### Security posture

- OpenSSF Scorecard instrumented; Dependabot + cargo-audit watching dependencies.
- Pinned GitHub Actions SHAs throughout CI.
- Daemon runs as the `wardnet` system user with only `CAP_NET_ADMIN` + `CAP_NET_RAW` + `CAP_NET_BIND_SERVICE`; systemd unit uses `ProtectSystem=strict`, `ProtectHome=yes`, `NoNewPrivileges=yes`, `PrivateTmp=yes`.
- Argon2id for admin password hashing, SHA-256 for session tokens.
- Parameterised SQL only — never string interpolation.

## Release signing

Starting with this release, every artefact is signed with [minisign](https://jedisct1.github.io/minisign/). The public key is committed to the repo at [`deploy/keys/wardnet-release.pub`](https://github.com/wardnet/wardnet/blob/main/deploy/keys/wardnet-release.pub) and will be embedded into the `wardnetd` binary in a future release for auto-update verification.

Each release ships three files per target:

- `wardnetd-<version>-<target>.tar.gz` — the stripped binary
- `wardnetd-<version>-<target>.tar.gz.sha256` — SHA-256 digest
- `wardnetd-<version>-<target>.tar.gz.minisig` — minisign signature

### Manual verification

```sh
# After downloading the three files:
minisign -Vm wardnetd-0.2.0-aarch64-unknown-linux-gnu.tar.gz \
  -p wardnet-release.pub \
  -x wardnetd-0.2.0-aarch64-unknown-linux-gnu.tar.gz.minisig
```

## Install

```sh
curl -sSL https://wardnet.network/install.sh | bash
```

Or clone and build from source — see the [development guide](https://github.com/wardnet/wardnet/blob/main/docs/DEVELOPMENT.md).

**Requirements:** a Linux host with `nftables`, `wireguard-tools`, `iproute2`, and `conntrack` available, and a LAN-facing network interface with permission to broadcast ARP.

## Targets in this release

Both targets are first-class; install whichever fits your network.

- `aarch64-unknown-linux-gnu` — Raspberry Pi (3 and newer) and other aarch64 Linux SBCs
- `x86_64-unknown-linux-gnu` — mini-PCs, x86_64 servers, and other non-Pi Linux hosts

## Known limitations

- **Auto-update not yet shipped on the daemon side.** This release establishes the signed-artefact pipeline; the in-daemon update runner, UI surface, and rollback unit land in v0.3.0.
- **Gateway resilience.** No active-passive failover, no hardware watchdog, no graceful shutdown hook. Plan a maintenance window if you upgrade.
- **TCP socket stickiness on tunnel switches.** After switching a device off a tunnel (VPN → direct, or between tunnels), existing TCP sockets may stay stuck for 30–60 s while the client-side stack times out. Routing on the Pi is correct immediately; toggling Wi-Fi on the device fixes it instantly. See [#77](https://github.com/wardnet/wardnet/issues/77).
- **NordVPN health signalling.** A server selected at tunnel-creation time may be unhealthy when you actually use it; the daemon currently marks such tunnels as "up" even when no handshake has completed. See [#79](https://github.com/wardnet/wardnet/issues/79) and [#80](https://github.com/wardnet/wardnet/issues/80).
- **Mobile app, kill switch per device, scheduled routing.** Planned but not in this release.

## Upgrading

This is the first tagged release, so there's nothing to upgrade from. Future daemon versions will carry forward the SQLite schema via sqlx migrations; this release introduces the versioned-release convention.
